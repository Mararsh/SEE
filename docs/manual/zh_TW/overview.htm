0 SEE是什麽

   Software Engineering Environment(SEE)是一個資訊處理框架。由sevlet+xml實現，通過瀏覽器使用。

   以下是SEE的幾個示例介面：

交叉圓形圖 交叉柱狀圖 比率柱狀圖 趨勢圖
個人消息 錯誤 "花"的介面

   安裝SEE後，用戶可以用"超級管理員"賬號（口令爲空）登錄，試一試各種功能。
   在"工作周報"和"問題報告"中包含了一些示例資料，可以直接用來試用統計圖表自動生成功能。

1 SEE簡介

   Software Engineering Environment(SEE)企圖提供軟體發展的資訊處理平臺。
   SEE是一個免費專案，即作者將同時將提供所有源代碼。
   它運行在任何支援servlet的環境下（作者只驗證了win和linux環境），沒用採用任何特定的實現技術，也不需要任何輔助程式。
   它的資料保存在XML中，不需要資料庫。（將來的版本會支援資料存儲的選項，這樣用戶可以選擇mysql這樣的免費資料庫或者商業資料庫，以提高處理速度）
   這意味著用戶可以零成本運行它。（tomcat是免費的servlet運行環境）
   用戶通過瀏覽器使用所有功能，因此資訊共用非常方便。
   SEE目前支援簡體中文、繁體中文和英語（只要有人翻譯任何語言都可支援）。在運行時，用戶可以隨時切換不同語言的介面。用戶亦可添加自己的文字翻譯，即時生效。
   SEE主體是資訊處理框架，其中所有資訊均可動態定制，而且即時生效。這意味著用戶可以隨意隨時修改、擴充自己的資訊結構。SEE提供了資訊屬性的多種預定義選擇，它們對應著多種預定義的資訊處理模式。定制者的關注點在描述資訊上，系統自動提供資訊的處理功能。這是SEE的核心。SEE的絕大多數功能都是這樣實現的。
   SEE缺省的資訊定義，是作者對於軟體工程資訊需求的理解。
   雖然SEE目標是軟體發展的資訊處理，實際上，用戶可以利用它設計處理任何類型的資訊（例如圖書登記、人事清單、財務報表等）。
   SEE的處理速度不差，20000條資料的讀取寫入不需1秒。目前排序速度是5000條9秒。（測試平臺CPU爲1.8G記憶體爲256M）

3 SEE版本

3.1 SEE最新版本

   目前SEE的版本號是0.07，這是第七個公開發佈的版本。這一版是爲了糾正致命的錯誤而發版的。

   各版本的變化請參見《更新記錄》。

   SEE的前幾個版本都將重點實現功能框架，而不是設計資訊結構。

   在版本1.0之前，用戶最好不要實際運用SEE。SEE的設計和實現都不穩定。

3.2 新變化

3.2.1 新增的功能
   這一版新增的功能有：

  • 調整資料介面，準備接入資料庫。
       這一版還未實現接入資料庫。

  • 設計實現新的介面風格"美麗的花"。
       這些花都是老婆婆今年在公園裏拍到的。
       老婆婆很喜歡這個叼花的小蜜蜂。這是老婆婆設計的第一個卡通形象。它將在你保存錯誤資料時出現。 ；）
       一個新介面風格就對應113*3張圖片！

  • 實現"個人消息"功能。
       用戶只能查詢自己發送和發給自己的消息、只能刪除發給自己的消息。
       用戶讀過一條發給自己的消息後，系統自動標記其爲"已讀"。
       介面上提示新收到的消息個數。
    3.2.2 增強的功能
       這一版增強的功能有：

  • 加強許可權管理的實現。
       對於"全部刪除"和"全部修改"，都是滿足於"基本條件"的情況下執行，即用戶只能全部刪除或者全部修改自己可以訪問的資料。
       只有有import許可權的用戶在導出資料時才可見password欄位，其他情況均不可見。
       所有權限表重大修改！關鍵字改爲"標題"。這樣更有利於定義多個條件。
       實現許可權表中"條件"的判斷。現在可是設置成任何合法的SEE條件。

  • 顯示圖片的縮略圖。
       在顯示資料時，對於文件類型的欄位值，若是圖片（jpg/jpeg/gif/png），則顯示其縮略圖。

  • 使資料比較時與語言相關，這樣各種語言的資料排序結果才正確。
       這個問題早就發現：中文排序結果沒有按照拼音順序。
    3.2.3 解決的問題
       這一版解決的問題有：（如此多的致命的、愚蠢的錯誤，作者爲此沮喪了好一陣子~~~）

  • 一般用戶不能訪問"用戶"表，因此很多資料都不能填寫！！
       這是一個致命的錯誤！這就意味著以前的所有版本都無法實際使用！
       現在許可權管理表重新定義了。

  • 新用戶加入時，若不設置密碼，則系統自己加上了密碼！這樣新用戶就無法登陸！
       這又是一個致命的錯誤！

  • 在上一版（0.06）中，英文版的系統許可權表居然是簡體中文的！
       這樣，英文版更是無法使用~~~
       這還是一個致命的錯誤！

  • 偏差處理裏的"風險項"定義中引用"風險分析"表名錯了。
       這就造成點擊"選擇"按鈕時彈出了錯誤資訊。

  • "口令"欄位可以通過查詢和導出看到。
       雖然是加密後的亂碼，但也會造成隱患！
       現在是：只有有import許可權的用戶在導出資料時才可見password欄位，其他情況均不可見。

  • 當關鍵字資料中包含引號時，頁面會有錯誤，功能按鈕（如修改、刪除）無法使用！
       這是因爲引號在頁面資料引用中造成錯誤。

  • 介面語言變化時，審計記錄的語言也變化了！
       這是不允許的。資料的語言是在安裝時已確定，不應隨介面語言變化！這個bug是因爲用錯了翻譯方法。
    3.2.4 資料的變化

       以下內容只對需要保留舊版本資料的用戶有用。

       這一版資料的變化有：

  • 添加"個人消息"。

  • 所有權限表重大修改！關鍵字改爲"標題"。

  • 給"不合格項"添加欄位"優先順序"。

  • 給"用戶"添加欄位"職務"。

       所有資訊結構的定義請參見"定制"一章。

    3.3 以後將實現的特點

  • 支援計算域的定義。
       計算域不是原始資料，其值不允許用戶輸入或修改，而是由其他資料根據即定公式自動得出。計算域常用於統計處理。

  • 提供更多的介面風格選擇。
       目前只有一個缺省的介面風格。介面風格的添加是"綠色的"，即只需拷貝css和相應的圖片到特定目錄下，系統自己就會發現，因此可以獨立提供給用戶下載。

  • 實現資料存儲方式的選擇。
       目前資料是保存在xml文件中。將來的版本允許用戶選擇將資料存儲在jdbc支援的任意資料庫中，連接參數在系統安裝時可配置。

  • 實現通用的工作流管理功能
       根據SEE自身的需求和框架，來實現實用的工作流機制。如果能兼顧國際標準更好。（難！）

  • 實現網路交流功能
       提供SEE自己的郵件管理、即時消息發送、論壇等功能。（工作量好大）

  • 設計實現軟體專案管理的大多數資訊處理。
       根據12207、cmm、國家標準的要求，分析設計資訊結構。這些模型中有很多重復/類似的資訊要求（與iso系列也有大量重復），系統只定義一套資訊結構，用戶可以切換不同的視角（介面方案）來進入這些模型。（難！）

  • 實現通用的文檔生成功能。
       系統根據文檔模板，由當前資料自動生成文檔。系統提供符合12207、cmm、國家標準的文檔模板，同時用戶可以自定義文檔模板。（難！）

       以上特徵，需要作者一個版本一個版本地來逐步實現。也許這些需求會有修改、刪除，也許會有更多的需求産生。

    3.4 上一版的特點

       截止上一版，SEE具有以下特色：

  • 免費。
       SEE自身免費，其運行環境均有免費的選擇。免費指源代碼可得。

  • 輕量的B/S模式。
       用戶端只需通用的瀏覽器，伺服器端只需servlet運行環境，SEE自身約6M（其中文檔和圖片爲5.2M）。

  • 資料相容。
       SEE中所有資料均是XML文件。而XML正是資料相容的産物。

  • 跨平臺。
       目前J2EE在多種作業系統上都有多種免費/商業産品。

  • 速度滿足實際應用。
       測試的結果表明：xml讀取寫入速度幾乎與資料量無關，當資料達到30000以上時，讀取寫入的耗時才一直處於1秒以上。
       SEE介面上有專門的按鈕，用來測試SEE的資料處理速度。每個資料處理頁面的右下角都會顯示當前頁面的回應速度。

  • 介面國際化和資料國際化。
       目前支援簡體中文、英文、繁體中文。
       用戶可以隨時切換到不同的語言介面。
       不同語言的安裝本版包含本地化的初始賬號和初始資料。在使用時，預製資料自動保存爲本地語言。

  • 用戶語言翻譯。
       現在用戶可以直接使用SEE來添加自己的語言文字翻譯。即時生效。
       這個功能對添加或修改資訊結構的用戶應該有用。

  • SEE的許可權管理機制。
       設置超級管理員，區分系統角色和資料角色，可以分層定義系統角色、資料角色和用戶的許可權。（這種分層許可權管理機制，參見使用指南的詳細說明）
       安裝後，系統中即有缺省的許可權資料。缺省的許可權模型基於這樣的思路：最大限度地開放專案資訊的管理能力，使專案資訊對專案組成員完全透明。在功能上，每個成員都可以任意添加/修改/查詢/審計專案的多數資料（刪除許可權只開放給資料管理員這個系統角色）。不必擔心有人"搗亂"，因爲所有操作行爲、資料內容和執行結果都可以被系統自動記錄。擁有充分的資訊，任何人都可以監督任何人（超級管理員不可見，她/他是系統的構建者）。
       當然，如果用戶需要採用其他許可權模型，完全可以刪除缺省資料、自己定義。

  • 用戶配置功能。
       超級管理員可以配置整個系統的參數，其她/他用戶可以設置即時生效的一些參數（介面語言、頁面大小、資料顯示風格、缺省專案等）。用戶的個人資訊中，還可以設置每次登錄都會生效的參數。

  • 通用的資訊結構定義功能。
       用戶可以隨時修改任何資訊結構定義，即時生效。用戶可以定義整個資訊結構的屬性（關鍵字組/關鍵字是否可修改/非空項/無效項/顯示項/缺省顔色等）和各個資料欄位的屬性（資料類型/大小/編輯寬度/單位/缺省值/值約束/編輯控制項類型等），系統根據資訊的定義來自動提供資訊的所有處理功能。這是SEE的核心。SEE的絕大多數資料和功能也是這樣實現的。即：定義系統與使用系統一樣，是通過運行SEE來實施的。

  • 通用的資訊處理基本操作。
       即：添加、複製、插入、修改、刪除。

  • 支援矩陣表。
       當一個資訊結構包含多個交叉的屬性時，使用矩陣表是最好的選擇。

  • 實現批量修改功能。
       用戶可以同時修改多個或者所有資料的某些欄位值。矩陣欄位可以被批量修改。
       在缺省許可權模型中，只有超級管理員和資料管理員可以執行批量修改操作。

  • 通用的資訊顯示設置功能。
       用戶可以任意設置資訊顯示的欄位、排序、顔色、風格，並且可以保存、修改和刪除這種顯示設置。

  • 通用的資訊查詢設置功能。
       用戶可以設置多個"而且"的條件關係，然後用"或者"的條件關係將它們連起來。條件運算元有：等於、不等於、大於、小於、開始于、結束於、包含、不包含。系統依據資料類型來比較資料。
       用戶可以保存、修改和刪除查詢設置。

  • 通用的資訊統計分析功能。
       目前實現計數、交叉計數和趨勢圖三種類型。用戶可以任意設置統計條件，系統自動生成統計報表（累計個數/百分率）、對比圖形（圓形圖/直條圖）和時間-資料趨勢圖 。
       用戶可以先設置查詢條件、再統計過濾後資料。
       用戶可以保存、修改和刪除統計設置。

  • 資料導出功能。
       用戶可以選擇導出所有或者當前頁面的資料，同時可以選擇導出的格式：xml、txt或html。這樣用戶可以利用其他工具來做更細緻的資料分析。同時利用顯示設置功能，用戶可以得到漂亮的查詢報告。
       用戶可以還選擇導出文件的編碼集，這樣有利於將資料導入其他工具。

  • 資料導入功能。
       用戶可以上傳一個格式正確的xml文件來導入資料。
       在缺省的許可權模型中，只有資料管理員和超級管理員可以導入資料。

  • 系統的審計功能。
       用戶的資料操作可以被系統自動記錄下來，包含：誰、何時、何地、什麽資料、原來的欄位值、新的欄位值、操作結果等。這樣，資訊的資料變化可以被完整地跟蹤下來。超級管理員可以設置需要審計的操作類型；用戶可以按資料或者按時間來查看審計記錄。這是專案監督的手段，也是資料恢復的一個途徑。
       用戶的登錄行爲也可以被審計。

  • 軟體專案常用的資訊結構。
       包括：專案跟蹤（進度/規模/成本）、軟體需求、軟體配置項登記、軟體評審、測試資訊、更動申請、問題報告、會議記要等。

  • 網路應用的基本設施。
       包括：登錄Cookie、口令加密、上傳管理等。

    4 誰可能對SEE感興趣

       以下理由可能會使讀者對SEE感興趣：

  • 你參與了一個軟體專案，專案組需要處理並共用：問題報告、測試結果、更動申請、會議記要、文檔登記、工作周報、工作日志之類的資訊。那麽SEE也許可以滿足你的需要；如果需要修改缺省的資訊定義，你不需要修改源碼，而是運行SEE來配置。

  • 你所在的組織需要一個工具來處理並共用某類資訊。運行SEE，添加一個新的資訊結構，也許在5分鐘內，你就可以實現你的想法。

  • 你想參考一個現成的資訊處理軟體系統。那麽SEE的使用模式和設計實現，也許可能有一點幫助

  • 你希望觀察一個現成的Sevelet應用。那麽SEE也許可以是個例子。

  • 你需要直觀瞭解各種軟體工程模型（CMM/12207/國家標準）的資訊要求。也許將來的SEE可以做出一點貢獻。

    5 SEE不做什麽

       SEE不打算成爲包羅萬象的軟體工程資訊處理平臺。
       例如，對於配置管理，利用免費的cvs或者商業軟體，也許更加有效。SEE至多可以提供資訊介面，以供資訊導入、導出、分析、查詢。
       又如，對於專案規劃和資料分析，使用唾手可得的現成工具（如project和excel），也許更爲合理。SEE至多提供登記規劃/分析結果的便利，以供專案組更容易地共用資訊。
